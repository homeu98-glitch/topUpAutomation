JWT field comments

FieldTypeWhat it is
issstringJWT issuer. Must be exactly site-a.
audstringJWT audience. Must be exactly site-b.
substringUnique external identity from Site A, for example owner:60000000 or customer:63936541.
jtistringUnique token id. Site B stores it and rejects reused tokens.
iatnumberIssued-at Unix timestamp in seconds.
nbfnumberNot-before Unix timestamp in seconds. Optional but recommended.
expnumberExpiry Unix timestamp in seconds. Controls how long the URL token stays valid.
rolestringowner or customer. Site B provisions the correct portal from this field.
membership_statusstringCustomer membership status from Site A. Use active for allowed customers.
shop.shopIdstring8-digit numeric Site A shop id. This is the main shop identity used by Site B.
shop.shopNamestringDisplay name of the shop from Site A.
customer.customerIdstring8-digit numeric customer/member id from Site A.
customer.displayNamestringCustomer display name from Site A.
customer.phonestringCustomer phone number if Site A wants Site B to store/show it.
redirect.pathstringTarget page inside Site B. Owner should use /owner.html, customer should use /.
redirect.txIdstring|nullOptional Site B transaction id for direct open of a rejected/failed transaction.

Site A → Site B flow

  1. Site A creates a JWT using HS256 and the SSO shared secret.
  2. Site A redirects the browser to the correct Site B entry URL with ssoToken in the query string.
  3. Site B verifies the JWT, auto-creates the owner/customer if missing, then writes its own secure session cookie.
  4. Site B removes the URL token from the address bar after login.
  5. While the user is active in Site B, authenticated API actions refresh the session expiry automatically.
The JWT is only for entry. After Site B verifies it, Site B switches to its own session cookie. Site A does not need to keep sending the JWT on every button click inside Site B.

Site B → Site A callback events

Site B signs all callback requests with these headers:

X-Topup-Event: <event name>
X-Topup-Timestamp: 2026-07-01T09:03:15.000Z
X-Topup-Signature: <hex hmac sha256 of "timestamp.body" using webhook shared secret>

Pending-count polling API

Site A can also actively poll Site B for the shop owner badge count.

GET https://top-up-automation.vercel.app/api/integration/sitea/pending-count?shopId=60000000
Authorization: Bearer <owner JWT signed by Site A with the same SSO secret>

Response

{
  "shopId": "60000000",
  "shopName": "表嫂美食",
  "pendingCount": 12,
  "updatedAt": "2026-07-01T09:06:20.000Z"
}
